# Explore 共享元素动画优化记录

## 背景

`/explore` 路由实现的是类似小红书、Pinterest 的卡片流交互：用户点击列表中的图片卡片后，图片从原位置放大为详情弹层，同时 URL
从 `/explore` 同步到 `/explore/$id`。

这个效果看起来像“共享元素动画”，但在当前项目里没有直接使用 `motion` 的 `layoutId` 作为主方案。原因是该项目启用了 React
Compiler，并且路由切换会影响组件生命周期。为了让动画在开发和生产环境都稳定，当前采用的是“URL 驱动 + Overlay 同层渲染 + 手动
FLIP 测量”的方案。

## 本次优化目标

本次优化不改变视觉效果和交互行为，重点降低后续维护成本：

- 拆分过大的 `useExploreOverlayState`。
- 移除关闭动画中的硬编码 `setTimeout(420)`。
- 减少卡片列表在打开详情时的无意义重渲染。
- 删除价值较低的中间 hook。
- 为关键状态机和测量逻辑补充注释。

## 修改内容

### 1. 拆分状态职责

原来的 `useExploreOverlayState` 同时处理：

- URL 中的详情 ID 解析。
- 卡片 DOM 节点注册。
- `getBoundingClientRect()` 测量。
- Overlay 目标布局计算。
- Body 滚动锁定。
- 打开、关闭、ESC 关闭。
- 关闭动画完成后的清理。

现在拆成了几个更小的模块：

| 文件                                                      | 职责                          |
|---------------------------------------------------------|-----------------------------|
| `src/features/explore/hooks/useExploreOverlayState.ts`  | 编排路由状态和 Overlay 动画状态        |
| `src/features/explore/hooks/useExploreCardRegistry.ts`  | 注册卡片图片节点，并读取起始矩形            |
| `src/features/explore/hooks/useExploreOverlayLayout.ts` | 根据 viewport 计算 Overlay 目标布局 |
| `src/features/explore/hooks/useBodyScrollLock.ts`       | Overlay 打开时锁定页面滚动           |
| `src/features/explore/types.ts`                         | 共享动画几何类型                    |

这样主 hook 的复杂度下降，后续如果要修改目标布局或滚动锁，不需要理解整套路由动画状态机。

### 2. 用动画完成事件替代定时器

之前关闭动画依赖 `setTimeout(420)` 清理 `overlayItem`。这种写法的问题是动画时长一旦调整，清理时机就可能和真实动画脱节。

现在由 `ExploreOverlay` 监听 `motion.div` 的 `onAnimationComplete`：

```tsx
onAnimationComplete = {()
=>
{
	if (isClosing) {
		onCloseAnimationComplete()
	}
}
}
```

关闭状态下，动画完成后再清理 Overlay 数据。这样动画时长变化不会破坏状态同步。

### 3. 减少卡片重渲染

点击卡片时，`RouteComponent` 会因为 `hiddenId`、`originRect`、`isClosing` 等状态变化而重渲染。这个父级重渲染不可完全避免，但可以阻止它扩散到所有卡片。

本次做了两点：

- `ExploreCard` 使用 `memo`。
- `handleOpen`、卡片点击回调、图片 ref 回调使用稳定引用。

优化后，列表父组件仍会重渲染，但未发生 props 变化的卡片可以跳过渲染。通常只有当前打开的卡片和上一个隐藏状态变化的卡片需要更新。

### 4. 删除 `useExploreCard`

原来的 `useExploreCard` 只包装了：

- 点击时调用 `onOpen(item)`。
- ref 时调用 `onImageRef(item.id, node)`。
- 根据 `hiddenImage` 返回 `visibility`。

这层抽象没有承载复杂逻辑，反而增加文件跳转成本。现在这些逻辑直接留在 `ExploreCard` 中。

## 为什么没有引入 Zustand

这次没有使用 Zustand，原因是它不能直接解决当前的核心问题。

当前性能风险主要来自 React 渲染边界和 props 稳定性，而不是“状态放在哪里”。如果把 Overlay 状态搬到 Zustand：

- 可以减少 `ExploreOverlay` 和 `ExploreCardList` 的 props drilling。
- 但仍需要维护 URL 与 store 的同步。
- 仍需要保存 DOM 测量结果。
- 仍需要处理关闭动画完成后的清理。
- 如果 selector 设计不好，依然会造成不必要重渲染。

因此当前阶段更适合使用局部 hook + 组件 memo。只有当 `/explore` 的 Overlay 状态需要被多个远距离组件共享，或者 props
drilling 明显继续恶化时，再考虑引入 Zustand。

## 当前方案边界

当前方案仍然不是“最简单”的共享元素动画，因为它手动维护了 FLIP 过程中的几何数据：

1. 点击前测量源图片区域。
2. 保存为 `originRect`。
3. 设置 `overlayItem` 让 Overlay 挂载。
4. 计算 `targetLayout`。
5. 使用 Motion 从 `originRect` 动画到 `targetLayout`。
6. 关闭时反向动画回 `originRect`。
7. 动画完成后清理本地 Overlay 状态。

这套方案的优势是稳定、可控、规避 `layoutId` 在路由和 React Compiler 下的不确定性。代价是代码量会多一些。

## 后续可继续优化

优先级从高到低：

1. 使用 React Profiler 验证卡片列表重渲染是否已经足够少。
2. 将 `ExploreOverlay` 的多个 props 封装成一个 `overlay` 对象，减少调用处噪音。
3. 给 `/explore/$id` 增加无来源卡片时的降级详情展示，改善直接访问详情 URL 的体验。
4. 如果卡片数量大幅增加，引入虚拟列表或分页，而不是继续只优化单个卡片渲染。
5. 如果 Overlay 状态开始跨多个页面共享，再评估 Zustand。

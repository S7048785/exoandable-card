# 项目核心功能原理

## 项目定位

这是一个 TanStack Start 全栈 React 应用。它使用 TanStack Router 做文件路由，使用 TanStack Query 作为数据请求和缓存层，使用
Vite 构建，使用 Tailwind CSS v4 和 CSS 变量管理样式主题。

应用目前的核心功能包括：

- 基础页面路由。
- 全局布局和主题初始化。
- TanStack Query 集成。
- `/posts` 示例共享元素动画。
- `/explore` 仿小红书瀑布流和详情 Overlay 动画。

## 路由系统

路由定义位于 `src/routes/`。项目使用 TanStack Router 的文件路由：

| 文件                             | 路由               | 作用                               |
|--------------------------------|------------------|----------------------------------|
| `src/routes/__root.tsx`        | 根路由              | 全局布局、Header、Footer、Devtools、主题脚本 |
| `src/routes/index.tsx`         | `/`              | 首页                               |
| `src/routes/about.tsx`         | `/about`         | 示例页面                             |
| `src/routes/posts.tsx`         | `/posts`         | `layoutId` 动画示例                  |
| `src/routes/posts.$postId.tsx` | `/posts/$postId` | posts 详情子路由                      |
| `src/routes/explore.tsx`       | `/explore`       | 瀑布流列表和详情 Overlay 容器              |
| `src/routes/explore.$id.tsx`   | `/explore/$id`   | explore 详情 URL，占位路由              |

`src/routeTree.gen.ts` 由 TanStack Router 自动生成，不应该手动编辑。

## 根布局

`src/routes/__root.tsx` 是应用的根布局。它负责：

- 引入全局样式。
- 初始化主题。
- 渲染 Header、Footer 和路由 Outlet。
- 注入开发工具。

所有页面都会经过根路由，因此适合放置全局 UI 和一次性初始化逻辑。不适合把具体业务页面的状态放在这里，否则会扩大重渲染范围。

## 数据层

TanStack Query 的 `QueryClient` 在 `src/integrations/tanstack-query/root-provider.tsx` 中创建，并通过 `src/router.tsx` 注入
Router context。

使用原则：

- 服务端或远程数据优先通过 TanStack Query 管理。
- 页面本地 UI 状态保留在组件或自定义 hook 中。
- 不要把瞬时动画状态放进 Query cache。

当前 `/explore` 使用的是 `src/lib/data.ts` 中的静态数据，所以不需要 Query。

## 样式和主题

全局样式位于 `src/styles.css`：

- 使用 CSS 变量描述主题颜色。
- 使用 `:root` 和 `:root[data-theme="dark"]` 区分浅色和深色主题。
- Tailwind CSS v4 负责工具类。
- shadcn/ui 风格组件通过 CSS 变量适配主题。

开发页面时，应优先复用已有 CSS 变量，不要在组件中随意增加孤立的颜色系统。

## `/explore` 核心原理

`/explore` 的交互目标是让用户感觉“列表里的图片变成了详情弹层”。为了保持 URL 可分享，点击卡片后路由会变为 `/explore/$id`
；为了保持动画连续，列表页不会卸载，详情以 Overlay 的形式渲染在同一个页面树中。

核心数据流：

1. 用户点击卡片。
2. `ExploreCard` 调用 `handleOpen(item)`。
3. `useExploreOverlayState` 读取该卡片图片节点的 `getBoundingClientRect()`。
4. 保存起始位置为 `originRect`。
5. 设置 `overlayItem`，让 `ExploreOverlay` 挂载。
6. 调用 TanStack Router 导航到 `/explore/$id`。
7. `useExploreOverlayLayout` 根据 viewport 计算目标布局。
8. `ExploreOverlay` 使用 Motion 从 `originRect` 动画到目标布局。
9. 关闭时 URL 回到 `/explore`。
10. Overlay 反向动画回源卡片位置。
11. 动画完成后清理 `overlayItem` 和 `originRect`。

这是一种手动 FLIP 方案：

- First：测量源卡片图片区域。
- Last：计算 Overlay 目标区域。
- Invert：用 Motion 的 initial/animate 表示从源位置到目标位置。
- Play：播放打开或关闭动画。

## 为什么 `/explore/$id` 返回 `null`

`src/routes/explore.$id.tsx` 目前只用于注册 URL，不直接渲染详情页面。详情 UI 仍由父级 `/explore` 页面控制。

这样做的原因是共享元素动画需要列表卡片和 Overlay 同时存在于 DOM 中。如果切到完全独立的详情页，列表卡片会卸载，动画就失去起点。

这个设计的不足是：直接访问 `/explore/$id` 时，体验依赖父页面是否能找到对应卡片数据和布局。后续如果要增强可分享链接体验，可以给详情路由增加降级内容。

## `/posts` 与 `/explore` 的差异

`/posts` 更像一个 `motion layoutId` 示例，它用共享 `layoutId` 让卡片和详情之间自动过渡。这个方案代码更短，但对路由生命周期、React
Compiler、生产构建稳定性更敏感。

`/explore` 使用手动 FLIP，代码更多，但优势是：

- 不依赖 `layoutId` 跨路由匹配。
- 可以精确控制关闭动画。
- 可以隐藏源图片，避免背景重复显示。
- 更容易处理移动端和桌面端不同布局。

因此 `/posts` 更适合展示概念，`/explore` 更接近产品级实现。

## 状态管理原则

项目目前没有引入全局客户端状态库。建议遵循以下边界：

- 路由状态：放在 URL 中，例如 `/explore/$id`。
- 远程数据：放在 TanStack Query 中。
- 局部 UI 状态：放在组件或 feature hook 中。
- 跨页面共享 UI 状态：达到明确需求后再考虑 Zustand 等状态库。

对于 `/explore`，Overlay 状态虽然看起来复杂，但仍属于页面局部 UI 状态。当前用 feature hook 管理比引入全局 store 更合适。

## 开发注意事项

- 不要手动编辑 `src/routeTree.gen.ts`。
- 修改 `/explore` 动画时，优先保持 `originRect`、`targetLayout`、`isClosing` 三者语义清晰。
- 修改动画时长后，不需要同步定时器；清理由 `onAnimationComplete` 触发。
- 如果新增大量卡片，应优先考虑分页、虚拟列表或图片懒加载。
- 如果发现卡片重渲染明显，应先用 React Profiler 验证，再调整 memo 或组件边界。
